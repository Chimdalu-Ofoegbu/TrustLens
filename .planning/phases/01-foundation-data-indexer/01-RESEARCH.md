## Summary

The census CSV was fully analyzed empirically in this session. It contains exactly 272 data rows with 272 unique numeric ids (no duplicates, range 1406–4879 with gaps). The headline finding upgrades the locked shifted-rating rule: **whenever `positive` is empty, the `rating` cell is either empty (33 rows) or an exact string copy of the price number (146 rows) — zero exceptions across all 179 such rows.** Critically, this includes rows with rating `'5'`, `'3'`, and `'1'` that would pass a naive 0–5 plausibility check. Two shift variants NOT anticipated in CONTEXT.md were discovered: 4 rows carry a full paragraph description in the `positive` column (ids 2169, 3091, 3092, 4188), and 1 row (id 4137) carries a 417-char description in the `sold` column. A deterministic rating rule that handles all cases with zero false positives/negatives was validated against the full file (90 rated / 182 unrated).

The category keyword table was drafted, executed against all 272 real name+tagline rows, and iterated three times. The final ordered first-match-wins table produces: Market Data & Analytics 25.7%, Security & Trust 16.5%, Trading & DeFi 15.1%, Lifestyle & Health 11.0%, Social & News 9.9%, Developer Tools & Infra 8.1%, Sports & Prediction 6.2%, Creative & Media 5.5%, Other Services 1.8% (5 agents). No bucket swallows the catalog; Other is a small remainder; no agent lands empty.

**Primary recommendation:** Build `indexer/` as four pure modules (`parse.py`, `category.py`, `db.py`, `census.py`) plus `refresh.py` orchestrator; stdlib only (`csv`, `sqlite3`, `unicodedata`, `logging`, `argparse`); use the verified parsers and category table in this document verbatim as the starting implementation; every numeric fixture value needed for tests is tabulated below from the real file.

### Price column — complete inventory

| Variant | Count | Parse |
|---------|-------|-------|
| `"N.NN USDT"` plain (incl. `"1 USDT"`, `"50 USDT"`, `"0.00 USDT"`×8) | 239 | `float` of leading token; observed range 0.00–50.0 |
| Subscript-zero `"0.0₄15 USDT"` / `"0.0₅1 USDT"` | 5 | See expansion rule below |
| Empty string | 28 | `price_usdt = NULL`, `price_raw = ""` |

**Non-ASCII characters in the price column — exhaustive:** U+2084 SUBSCRIPT FOUR (`₄`, 1 occurrence) and U+2085 SUBSCRIPT FIVE (`₅`, 4 occurrences). No other subscript digits, no other non-ASCII. Parser must still accept the full U+2080–U+2089 range (locked decision; costs nothing).

**All 5 subscript rows:** 2023 Onchain Data Explorer `0.0₄15 USDT` → 0.000015; 1851 Ethy AI, 2088 SignalSmith, 2087 XBubbleAI, 1888 Hunch Research — all `0.0₅1 USDT` → 0.000001.

**Subscript semantics (off-by-one trap):** the subscript digit is the TOTAL count of zeros between the decimal point and the significant digits: `0.0₄15` = `0.` + `0000` + `15` = 0.000015. The displayed literal `0` after the point is stylistic prefix, NOT an additional zero. A first-draft parser written in this session made exactly this off-by-one (producing 0.0000015 — a 10× error). Regex `0\.0([₀-₉])(\d+) USDT` → `float(f"0.{'0'*sub}{digits}")` where `sub` = subscript value. Verified: both real variants produce the locked expected values.

**28 empty-price ids:** 2791, 3841, 2067, 4427, 2766, 2444, 3117, 3890, 3030, 3700, 3701, 3772, 2627, 2343, 2340, 3468, 2103, 2933, 3627, 2205, 2220, 4360, 2978, 2284, 4879, 4147, 3053, 3913. **Six of these are RATED agents** (2791, 3841, 2067, 4427, 2766, 2444 — all `5.0` / `100% positive` / 1–2 sold): empty price does NOT imply unrated. Parse price and rating independently.

### Rating & positive columns — the shifted-column artifact, fully mapped

Exact cross-tabulation of all 272 rows:

| `positive` cell | `rating` cell | `price` | Count | Correct storage |
|---|---|---|---|---|
| `"N%positive"` pattern | real value ≠ price token | present | 83 | rating=value, positive_pct=value |
| `"N% positive"` pattern | real value ≠ price token | empty | 6 | rating=value, positive_pct=value, price NULL |
| paragraph text | `5.0` ≠ price token `1` | present | 1 (id 2169 FundingArb) | rating=5.0, positive_pct=NULL |
| paragraph text | exact echo of price token | present | 3 (ids 3091, 3092, 4188) | rating=NULL, positive_pct=NULL |
| empty | **exact string echo of price token** | present | 146 | rating=NULL, positive_pct=NULL |
| empty | empty | present | 11 | rating=NULL, positive_pct=NULL |
| empty | empty | empty | 22 | all three NULL |

**The echo rule (zero violations in 272 rows):** when `positive` is empty, `rating` is either empty or a character-for-character copy of the price number (`"0.5"` when price is `"0.5 USDT"`, `"1"` when `"1 USDT"`, `"0.0001"` when `"0.0001 USDT"`). **This includes echoes that look like valid ratings:** `'5'` (ids 4489, 3319, 2249, 2732 — all priced 5 USDT), `'3'` (id 2411, 3 USDT), `'1'` (32 rows, 1 USDT), `'1.88'` (id 2082). A naive "rating must be 0–5" check would accept ~40 false ratings. Among genuinely rated rows, rating strings are always one-decimal-formatted (`2.0`–`5.0`); rating `0.x` never occurs with a valid positive.

**Recommended deterministic rule (validated, 0 false positives / 0 false negatives on the full file):**

> rating is valid **iff** `positive` is non-empty AND `rating` string ≠ price-number token AND `0 ≤ float(rating) ≤ 5`. positive_pct is valid **iff** it fullmatches `\d+(\.\d+)?% positive`.

Result: **90 rated / 182 unrated** (89 with positive_pct + FundingArb's 5.0 with positive_pct NULL). Verified: no clean row has rating-string == price-token, so nothing real is nulled. This is a strict superset of the locked rule ("price-like value AND positive empty → NULL") — it additionally resolves the 4 paragraph-positive rows CONTEXT didn't anticipate, in the same never-fabricate spirit. **Simpler alternative** (planner may choose): rating valid iff positive matches the % pattern → 89 rated, loses only FundingArb's plausible 5.0. Both satisfy the locked constraint; recommend the first.

**The 4 paragraph-positive rows** (2169 FundingArb, 3091 八字精批, 3092 塔罗占卜, 4188 Falsify): the `positive` cell holds the agent's FULL description (493–1245 chars; 4188's includes embedded newlines) and the `tagline` cell holds the first 150 chars of the same text. Treat as positive_pct=NULL; keep tagline as-is (consistent 150-char truncation across the catalog).

## Category Derivation — Final Table & Verified Distribution

Method: match against `unicodedata.normalize("NFKC", f"{name} {tagline}").casefold()`. Buckets checked in order; within a bucket, keywords in order; first match wins; no match → Other Services. ASCII keywords use `\b` word boundaries; CJK keywords and phrases are substring matches. Three iterations were run against all 272 real rows to converge (v1 issues found and fixed: DeFi "health factor" pulled loan tools into Lifestyle; "not astrology" negation pulled ChainAlmanac into Lifestyle; "funding rate"/"trading volume"/"perpetual prices" pulled data APIs into Trading; plural "cafes"/"audits" missed; `\bdapp\b` grabbed a wallet profiler; bare "content" grabbed a BTC trading agent that analyzes YouTube content).

### Anti-Patterns to Avoid

- **Range-checking rating without the positive gate:** `0 <= float(r) <= 5` accepts ~40 price echoes including four perfect-looking `'5'`s — the verified rule above is the only safe one
- **UNIQUE on name_key:** crashes on real data (two verified collisions)
- **`.encode('ascii','ignore')` / stripping non-ASCII anywhere near prices:** turns 0.000015 into 0.015 (1000×) [CITED: PITFALLS.md #8]
- **Line-based row handling:** 303 physical lines ≠ 272 rows; `wc -l` sanity checks are wrong for this file
- **`datetime` objects into sqlite3:** implicit adapters deprecated since Python 3.12 — pass ISO-8601 strings only [CITED: STACK.md, verified stdlib behavior]
- **Printing raw agent names in log/console messages on Windows dev:** cp1252 console raises UnicodeEncodeError on CJK (hit during this session) — log ids/row numbers; if text is needed, `ascii()`/`repr()` it or reconfigure stdout to UTF-8 in `main()`

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Quoted/multiline CSV records | split/regex line readers | stdlib `csv` with `newline=""` | 29 embedded LFs + 227 comma-taglines + 5 quote-taglines in the real file |
| Unicode name folding | manual char maps / ascii tricks | `unicodedata.normalize("NFKC", s).casefold()` | Folds U+FF1F and width variants correctly; verified on id 3345 |
| Upsert semantics | SELECT + branch INSERT/UPDATE | `INSERT ... ON CONFLICT DO UPDATE` | Atomic, one statement, `excluded.` pseudo-table preserves first_seen cleanly |
| Category assignment | LLM/embedding classification | in-code ordered keyword table | Determinism is a locked requirement; table is testable and diffable |
| Timestamps | wall-clock at call sites | injected `captured_at` parameter | Locked; keeps every test byte-stable |

## Common Pitfalls

### Pitfall 1: The rating echo that looks like a 5-star rating
**What goes wrong:** ids 4489/3319/2249/2732 have rating cell `'5'` with 0 sold — naive parsing stores a perfect rating that is actually the 5 USDT price echoed.
**Why it happens:** the census capture writes the price into the rating slot for unrated agents; 0–5 range checks pass.
**How to avoid:** the verified rule — rating valid iff positive non-empty AND rating ≠ price token AND in range.
**Warning signs:** any stored rating where `positive_pct IS NULL` and rating == price_usdt; count(rated) ≠ 90.

### Pitfall 2: Subscript expansion off-by-one
**What goes wrong:** `0.0₄15` parsed as "0.0" + 4 zeros + "15" = 0.0000015 (10× low). Made in this session on first attempt.
**Why:** the displayed literal `0` after the point is part of the compressed display; subscript = TOTAL zeros.
**How to avoid:** `float(f"0.{'0'*sub}{digits}")` with regex `0\.0([₀-₉])(\d+)`; fixture-test both real values (0.000015, 0.000001).
**Warning signs:** any parsed price < 1e-06 (real minimum is 0.000001) or in (0, 1e-06).

### Pitfall 3: sold column containing prose (id 4137)
**What goes wrong:** `int()` raises, or digit-grabbing regexes extract garbage (e.g., "21" from "X Layer" text... any digits in the paragraph).
**How to avoid:** `re.fullmatch(r"([\d,]+(?:\.\d+)?)([KMkm])? sold", cell)`; no match → sold=0 + `logging.warning` with row number and id.
**Warning signs:** sold values that don't appear in the raw file; more than 1 sold-warning on the census.

### Pitfall 4: Paragraph text in `positive` with embedded newlines (id 4188)
**What goes wrong:** `cell.endswith("% positive")` or split-based parsing misfires; the 1245-char cell for 2169 even contains the `%` character in prose contexts elsewhere — only a strict fullmatch is safe.
**How to avoid:** `re.fullmatch(r"(\d+(?:\.\d+)?)% positive", cell.strip())`.

### Pitfall 5: Assuming price present ⇒ ratable / price empty ⇒ junk row
**What goes wrong:** 6 rated agents (5.0/100%) have EMPTY price; 22 rows have rating+positive+price all empty but valid sold counts. Coupling field parsers corrupts good fields.
**How to avoid:** each field parses independently; a row is never skipped wholesale (id/name/tagline are always present — verified 0 empties in those columns).

### Pitfall 6: Windows dev-console encoding
**What goes wrong:** `print(name)`/log handler raises `UnicodeEncodeError: 'charmap' codec...` on CJK names under cp1252 (reproduced in this session).
**How to avoid:** log ids not names, or `sys.stdout.reconfigure(encoding="utf-8")` in the CLI entry; never let a log statement crash the refresh.

### Pitfall 7: Category table drift breaking determinism
**What goes wrong:** ad-hoc keyword edits during later phases silently re-bucket agents, changing Phase 2 percentiles and leaderboard output.
**How to avoid:** a test pins the full 272-row distribution (counts per bucket) and the fixture rows' categories; any table edit must consciously update the snapshot.

## Code Examples

### parse_sold
```python
# Source: verified against all 272 real rows this session
import re
_SOLD = re.compile(r"([\d,]+(?:\.\d+)?)\s*([KMkm])?\s*sold")

def parse_sold(cell: str) -> int | None:
    """'539 sold'->539, '1.55K sold'->1550; None if unparseable (caller stores 0 + warns)."""
    m = _SOLD.fullmatch(cell.strip())
    if not m:
        return None                       # real case: id 4137 carries prose here
    n = float(m.group(1).replace(",", ""))
    mult = {"K": 1_000, "M": 1_000_000}.get((m.group(2) or "").upper(), 1)
    return int(round(n * mult))
```

### parse_price
```python
# Source: verified against all 272 real rows; subscript semantics per locked CONTEXT rule
_SUB = {chr(0x2080 + i): i for i in range(10)}          # U+2080..U+2089 -> 0..9
_PLAIN = re.compile(r"(\d+(?:\.\d+)?)\s*USDT")
_SUBP  = re.compile(r"0\.0([₀-₉])(\d+)\s*USDT")

def parse_price(cell: str) -> float | None:
    s = cell.strip()
    if not s:
        return None                                     # 28 real rows
    if (m := _SUBP.fullmatch(s)):
        return float(f"0.{'0' * _SUB[m.group(1)]}{m.group(2)}")  # 0.0₄15 -> 0.000015
    if (m := _PLAIN.fullmatch(s)):
        return float(m.group(1))
    return None                                         # unseen format -> NULL + warning
```

### parse_rating_positive (the echo rule)
```python
# Source: validated on the full file — 90 rated / 182 unrated, zero misclassifications
_PCT = re.compile(r"(\d+(?:\.\d+)?)% positive")

def _price_token(price_cell: str) -> str:
    s = price_cell.strip()
    return s[:-len(" USDT")].strip() if s.endswith(" USDT") else s

def parse_rating_positive(rating_cell: str, positive_cell: str,
                          price_cell: str) -> tuple[float | None, float | None]:
    rt, pos = rating_cell.strip(), positive_cell.strip()
    pct = float(m.group(1)) if (m := _PCT.fullmatch(pos)) else None
    rating = None
    if pos and rt and rt != _price_token(price_cell):   # echo gate (146+3 real echoes)
        try:
            v = float(rt)
            rating = v if 0.0 <= v <= 5.0 else None
        except ValueError:
            rating = None
    return rating, pct
```

import unicodedata
